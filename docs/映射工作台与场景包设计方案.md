## 〇、目标与故事线

解决三个已确认的功能缺口：

1. **场景单一** → 报表定义从代码变成数据（场景包），新增报表零代码
2. **注释不清晰无法判断映射** → 多通道证据融合的映射推断引擎，置信度分级
3. **无可视化映射干预** → 映射工作台 + 编排器 human-in-the-loop 暂停/确认/恢复

**演示故事线**："业务方自助配置新报表（场景包）→ AI 多通道推断字段映射（附证据）→ 专家在工作台确认/修改（全程留痕）→ 确认结果沉淀为历史映射资产 → 下次同类任务自动复用"

## 二、后端服务设计

### 2.1 场景包服务（services/report_pack_service.py）

- CRUD + 种子（`scripts/seed_report_packs.py`，幂等）
- **内置 3 个场景包**：
  - `G01`：1104 资产负债项目统计表·个人住房贷款（现有场景迁移）
  - `G11`：1104 资产质量五级分类（源表 loan_contract，重点：五级分类映射、逾期 90 天）
  - `EAST_JJ`：EAST 信贷业务借据表（现有 EAST 场景迁移）
- Agent 1/2 改造：`regulation_parser` 和 `codegen` 从 report_pack 读取目标结构/候选源表/勾稽规则/检索关键词，替换硬编码；任务创建时可指定 report_pack_id（缺省 G01 保持兼容）

### 2.2 数据画像服务（services/profiling_service.py）

- `profile_column(table, column)` → `{null_rate, distinct_count, sample_values(≤10), min/max(数值), format_pattern(正则: 证件号/手机号/日期/金额), enum_values(低基数时)}`
- 走 database_mcp 只读通道执行统计 SQL（复用 sqlglot 纵深，仅聚合查询）
- 结果按 (table, column) 缓存

### 2.3 映射推断引擎（services/mapping_engine.py）

```python
def infer_mappings(report_pack, schemas) -> list[FieldMapping]:
    # 对每个 target_field，对每个候选 (source_table, source_field) 计算五通道得分：
    # 1. name      名称相似度：编辑距离/拼音/缩写词典(amt→金额,bal→余额,no→编号,days→天数)
    # 2. comment   注释语义：BGE 余弦（注释缺失则该通道 None，不计入权重）
    # 3. profile   数据画像：类型兼容 + 值域/枚举与 expected_domain 匹配
    # 4. semantic  制度语义：caliber_text ↔ (字段名+注释+画像摘要) 的 BGE 余弦
    # 5. history   历史映射库命中：1.0（命中）/ None（未命中）
    # 融合：可用通道加权平均（权重可配 name.2/comment.2/profile.3/semantic.2/history.1）
    # 分级：≥0.85 ai_inferred(高置信) / 0.5-0.85 待确认 / <0.5 unmapped
```

### 2.4 human-in-the-loop 编排改造（core/orchestrator.py）

```
regulation_parser → mapping_inference（新阶段，轻量不入 Agent 序列，作为阶段间步骤）
  ├─ 全部 ≥0.85 且任务 auto_mode=True → 直接进 codegen（演示路径不变）
  └─ 否则 → status=waiting_confirmation，checkpoint={"next":["codegen"],"pause_reason":"mapping_confirmation"}
确认 API 被调用 → 校验全部 target_field 有终态（confirmed/modified/needs_etl 不算阻断，unmapped/rejected 必须处理）
  → status=queued，worker 从断点续跑 codegen（codegen prompt 注入已确认映射）
```

- `codegen` 改造：prompt 注入 confirmed mappings（"以下映射已由专家确认，必须采用"）
- 映射确认后写入 mapping_assets（复用计数 +1）

### 2.5 API 设计（契约）

```
# 场景包（api/report_packs.py）
GET    /v1/tenants/{tid}/report-packs
GET    /v1/tenants/{tid}/report-packs/{pack_id}
POST   /v1/tenants/{tid}/report-packs            # admin
PUT    /v1/tenants/{tid}/report-packs/{pack_id}  # admin

# 映射（api/mappings.py）
GET    /v1/tenants/{tid}/tasks/{task_id}/mappings            # 映射清单（含 evidence/画像）
POST   /v1/tenants/{tid}/tasks/{task_id}/mappings/{mid}/confirm   # {transform_rule?} 可改
POST   /v1/tenants/{tid}/tasks/{task_id}/mappings/{mid}/modify    # {source_table, source_field, transform_rule}
POST   /v1/tenants/{tid}/tasks/{task_id}/mappings/{mid}/reject    # 拒绝 AI 推断
POST   /v1/tenants/{tid}/tasks/{task_id}/mappings/{mid}/needs-etl # 标记需 ETL 加工
POST   /v1/tenants/{tid}/tasks/{task_id}/mappings/confirm-all     # 全部确认 → 任务恢复 queued
GET    /v1/tenants/{tid}/mapping-assets                          # 历史映射资产库

# 画像（并入 mappings 响应，不单独开 API）
# 全部挂现有鉴权 + 审计（mapping.confirm / mapping.modify / report_pack.create 等动作）
```

## 三、前端设计（第 6、7 页面）

### P6 映射工作台（pages/MappingWorkbench.tsx）

- 入口：任务大厅/执行页中 `waiting_confirmation` 任务高亮"待确认映射"按钮
- 布局：左列目标字段（含 caliber_text 提示）、右列 AI 推断的候选源字段（连线：实线高置信/虚线待确认/红色未映射）
- 点击映射 → 底部详情面板：五通道证据条形、源字段画像（空值率/枚举/样例值）、转换规则编辑框、[确认][修改][拒绝][需ETL]
- 顶部：进度"已确认 3/8"、[全部确认并继续执行]（全部有终态才可点）
- 无连线图库依赖：CSS/SVG 自绘，Demo 深度

### P7 场景包管理（pages/ReportPacks.tsx）

- 场景包列表（代码/名称/类型/状态/目标表）+ 详情抽屉（目标结构/勾稽规则/关联制度）
- 新建/编辑（admin）：表单 + target_schema JSON 编辑器（AntD 表格编辑即可）
- 新建任务弹窗：增加"选择场景包"下拉（替换原硬编码模板）

### 任务大厅/执行页适配

- `waiting_confirmation` 状态金色"待确认"Tag + 跳转工作台按钮
- 执行页阶段流中显示"映射确认"节点（人工节点样式区分）

## 四、开发任务拆解（并行四路）

| 路 | 范围 | 关键产出 | 文件所有权 |
|---|------|---------|-----------|
| **A 场景包** | models/report_pack.py、services/report_pack_service.py、api/report_packs.py、scripts/seed_report_packs.py、Agent1/2 读包改造、tests | 3 个内置包、CRUD API、Agent 包驱动 | models/__init__.py 归 A |
| **B 映射引擎** | models/field_mapping.py、models/mapping_asset.py、services/profiling_service.py、services/mapping_engine.py、tests | 五通道推断、画像服务、置信度分级 | 不动 models/__init__.py（A 统一加） |
| **C HITL 编排** | core/orchestrator.py、api/mappings.py、api/tasks.py（auto_mode）、codegen prompt 注入映射、main.py（注册全部路由）、tests | waiting_confirmation 状态机、确认 API、断点恢复衔接 | main.py 归 C |
| **D 前端** | pages/MappingWorkbench.tsx、pages/ReportPacks.tsx、App.tsx 路由/菜单、任务大厅/执行页适配、api/client.ts 类型 | P6/P7 页面 | frontend/ 归 D |

**依赖协调**：B/C/D 都依据本文档的数据模型与 API 契约开发，不等彼此完成；整合由协调者统一做（含 import 修正、回归、提交）。

**测试要求**：每路自带 pytest（A：包 CRUD+种子；B：五通道打分/分级/画像；C：暂停-确认-恢复全链路+自动模式不暂停）；前端 npm run build 通过。

**验收标准**：
1. 用 G11 场景包创建任务 → Agent 1 读包检索 → 映射推断产出（G11 的"五级分类"应对 loan_contract 某字段高置信，故意缺注释字段走画像通道）→ 任务挂起 waiting_confirmation
2. 工作台确认/修改映射 → confirm-all → 任务从 codegen 续跑至 completed
3. 确认后的映射进 mapping_assets；再次创建同包任务时 history 通道命中
4. auto_mode=True 且全高置信时不暂停（回归现有演示路径）
5. 全套 pytest + 三冒烟回归绿
