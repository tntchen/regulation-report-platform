# 零贷监管报表平台 — 方案评审与改进路线图

> **版本**：v1.0
> **日期**：2026-07-23
> **评审范围**：融合系统架构设计、Demo v3 规划、M1–M4 落地代码（commit `1fb8d07`）
> **评审视角**：监管 / 银行业务 / 科技架构 / 研发 / 测试 / 运维
> **代码证据**：全部问题定位到 文件:行号，可逐条复核

---

## 〇、总体结论

当前系统（M1–M4 完成后）是一个**精心编排的演示闭环**：Mock AI 生成 SQL → 六维规则自检 → SQLite 种子数据双口径对账 → 产出 4 份交付文档。链路完整、演示效果好，6Agent 协作模式的可行性已得到验证。

但用"平台"的标准衡量，**认证、鉴权、审计、异步任务、SQL 方言、真实数据源、真实向量检索——七根支柱均未建立**。当前正确定位是：

> **"验证了 6Agent 协作模式可行性的 PoC"，而非平台。**

把它叫平台，评审方会替我们指出差距；自己先声明边界，差距就转化为"规划清晰的演进路线"。

## 一、六视角评审

### 1.1 监管视角 ⚖️

**核心问题：监管最看重的"留痕"与"权限"恰好为零。**

| # | 发现 | 证据 | 严重度 |
|---|------|------|--------|
| R1 | 全后端无一个 `logging` 调用，任务创建 / SQL 执行 / 文档操作均无审计记录，`uploaded_by` 硬编码 `"api"` | `backend/` 全局；`api/regulations.py` | **blocker** |
| R2 | `get_tenant` 仅校验租户存在性、无任何身份认证；URL 换 `T001→T002` 即可访问另一租户全部数据 | `api/deps.py:10-16` | **blocker** |
| R3 | `list_tasks`/`get_task` 未挂租户校验依赖，任意租户 ID 可枚举任务 | `api/tasks.py:46-67` | **blocker** |
| R4 | 前端下拉框自由切换租户即完成"身份切换"，越权被做成了产品功能 | `frontend/src/App.tsx:27` | **blocker** |
| R5 | 平台自身不执行自己制度库里的脱敏要求（C2/C3 字段），日志/API 返回/向量库内容均无脱敏 | `agents/quality_gate.py:316-325`（仅对生成 SQL 文本做 warning 级正则） | major |

**监管视角结论**：越权与无留痕在银行语境下不是缺陷、是事故。即使 Demo 阶段，也应补"演示级 RBAC + 审计日志"，并把操作留痕做成可演示的功能点（反而是加分项）。

### 1.2 银行业务视角 🏦

**核心问题：业务闭环是"自证预言"，换一个场景立即露馅。**

| # | 发现 | 证据 | 严重度 |
|---|------|------|--------|
| B1 | Mock 返回写死的住房贷款 SQL，与输入 prompt 几乎无关（仅正则提取两个表名） | `core/ai_adapter.py:100-171` | major |
| B2 | Agent 1 的"口径提取/陷阱识别"是关键词字符串匹配，不经 AI；字段映射建议返回空列表 | `agents/regulation_parser.py:68-104`、`agents/codegen.py:136-139` | major |
| B3 | database_mcp 的 schema 只硬编码 2 张表、execute_sql 返回固定 5 行假数据——Agent 2 实际只"见过" 2 张表 | `mcp/database_mcp.py:26-101` | major |
| B4 | 数字孪生仅"1104 vs EAST 贷款余额"单一场景；制度变更预演、假设分析未做 | `agents/digital_twin.py` | major |
| B5 | T002（Oracle）租户配置存在但引擎工厂不支持 oracle，切换到 T002 跑真实链路必失败 | `database.py:67-71`、`core/tenant_context.py:71-154` | major |
| B6 | AI api_key 为空时静默降级 Mock，配错 Key 不报错而是悄悄跑假数据——Demo 最危险行为 | `core/ai_adapter.py:210-211` | **blocker**（生产化必须 fail-fast） |

**业务视角结论**：需要至少 3 个报表场景的真实链路证明泛化性；Mock 静默降级必须改为 fail-fast；孪生应补"制度版本预演"这一业务方真正心动的场景。

### 1.3 科技架构视角 🏗️

**核心问题：架构图画了六层，代码里只有一层半。**

| # | 发现 | 证据 | 严重度 |
|---|------|------|--------|
| A1 | 创建任务是 HTTP 请求内同步跑完 6 Agent（真实 AI 下分钟级），无队列、无后台 worker、无超时控制 | `api/tasks.py:32-43` | **blocker** |
| A2 | 编排器是内存状态机、无持久化断点：进程重启后 `executing` 任务永远成僵尸，无恢复/续跑/清理 | `core/orchestrator.py`、`api/tasks.py:33-34` | **blocker** |
| A3 | 生成 MySQL 方言 SQL 却在 SQLite 上验证（无类型列建表绕过方言差异），"可投产 SQL"核心承诺未验证 | `agents/test_verify.py:131` | **blocker** |
| A4 | embedding 是 md5 拼 768 维伪向量，且检索根本不用向量（纯 bigram 文本打分）——"向量库"名不副实 | `services/vector_service.py:69-80,179-195` | **blocker**（须声明为 Demo 裁剪） |
| A5 | 向量索引单 JSON 全量读改写、无锁，并发索引/删除必丢数据 | `services/vector_service.py:96-120` | major |
| A6 | 同步 SQLite 驱动在 async 代码中直接调用，阻塞事件循环；每次查询新开连接无池化 | `mcp/demo_dataset.py:91-99` | major |
| A7 | 禁用文档不删向量、靠运行时过滤，但 Agent 1 走的 `regulation_rag.retrieve` 不传 `active_doc_ids`——被禁用文档仍参与检索（一致性真实 bug） | `mcp/regulation_rag.py:140-144`、`api/regulations.py:213-224` | major |
| A8 | 门禁回退不重跑 Agent 1（检索结果可能过期）、无幂等键（重复提交生成多个报送任务） | `core/orchestrator.py:155-178` | major |

**架构视角结论**：任务模型（异步化+断点恢复）、方言验证、真实向量是三个必须重建的地基；并发与一致性问题需要锁与事务补齐。

## 二、Demo 边界声明（L0，应立即补入 README）

以下 10 点是 Demo 合理裁剪，但此前未在 README 声明，评委/生产方一看就会问：

1. database_mcp 不执行真实 SQL（schema 硬编码 2 张表、返回固定假数据），"只读红线"是演示桩
2. 向量是 md5 伪向量且检索不用向量，实际是关键词 bigram 检索；faiss/embedding 配置是空头支票
3. 质量门禁自检自证：Mock SQL 是按门禁规则"量身定做"的
4. 数字孪生/测试验证的"真实数据"是 12 行 SQLite 种子，Agent 4 对账口径硬编码
5. T002（Oracle）租户无法真实运行
6. 无认证/多租户自由切换，"多租户"演示的是数据分目录而非权限体系
7. Docker `--reload` + 数据打入镜像 + 明文密码，是开发编排非部署方案
8. Agent 1 口径提取是关键词匹配不经 AI，字段映射表来自硬编码文本
9. 任务"持久化"只持久化状态不支持恢复，重启后 executing 任务成僵尸
10. platform.db / 租户目录无备份、无迁移工具

## 四、L2 平台化改进计划（2 周）

> **目标**：从 PoC 演进为"配得上平台二字"的系统——有权限、有留痕、有真实数据、有真实检索、任务可恢复。
> **原则**：每个工作日结束都有可验证产出；每项改动配测试；全程保持冒烟回归绿。

### Week 1：安全与可靠性地基

#### Day 1–2：认证 + 租户鉴权（P0）

| 任务 | 内容 | 验收标准 |
|------|------|---------|
| 用户与认证 | users 表 + 登录 API（JWT，HS256）；`secret_key` 从环境注入无默认值；密码 bcrypt 哈希 | 无 token 调 API 返回 401；错误密码 401；token 过期 401 |
| RBAC + 租户绑定 | 用户-租户多对多绑定表；`get_tenant` 升级为"认证 + 租户成员校验"；全部 API 挂鉴权依赖（含 tasks.py 补漏） | 用户 A 访问未授权租户返回 403；`list_tasks` 不再能枚举他租户 |
| 前端登录 | 登录页 + token 存储 + axios 拦截器（401 跳登录）；租户切换器只显示本用户有权限的租户 | 浏览器全流程可走通 |
| 测试 | 认证/越权 API 集成测试（TestClient） | 越权用例全红转绿 |

#### Day 3：审计日志（P0）

| 任务 | 内容 | 验收标准 |
|------|------|---------|
| 审计表与中间件 | audit_logs 表（who/when/tenant/action/resource/detail/ip）；FastAPI 中间件自动记录全部写操作；Agent 关键动作（SQL 执行、文档上传/删除/索引、任务创建/完成/失败）手动埋点 | 每个写操作都有审计记录 |
| 审计查询 API | GET /v1/tenants/{tid}/audit-logs（分页+过滤） | 前端可查 |
| 全局 logging | 引入 logging 配置（结构化、按级别），关闭 SQL echo；错误带 trace_id | 日志文件可查，无敏感字段明文 |
| 演示价值 | 前端"审计留痕"入口（可放向量库维护页或独立页） | 演示脚本可展示"谁上传了文档、谁建了任务" |

#### Day 4–5：任务异步化 + 断点恢复（P0）

| 任务 | 内容 | 验收标准 |
|------|------|---------|
| 后台任务队列 | 创建任务改为"落库 queued → 立即返回 task_id"；内置 asyncio worker（DB 轮询取任务执行），支持并发上限（启用 tenant.max_concurrent_tasks） | POST 秒回；任务在后台推进；前端轮询可见状态流转 |
| 断点恢复 | 每个 Agent 阶段完成即落库断点；worker 重启后扫描 `executing/queued` 任务：queued 重新执行、executing 从最后完成阶段的下一阶段续跑 | kill -9 后重启，任务从中断点续跑完成 |
| 幂等与取消 | 创建接口支持幂等键（client_request_id 唯一约束，重复提交返回已有任务）；POST /tasks/{id}/cancel 取消执行中任务 | 重复提交不产生重复任务；取消后状态 cancelled |
| 回退修正 | 门禁回退时重跑 regulation_parser（刷新检索上下文）或显式声明缓存理由 | 回退链路测试更新 |

### Week 2：数据与 AI 真实化

#### Day 6–7：database_mcp 真实执行 + SQL 安全纵深（P0）

| 任务 | 内容 | 验收标准 |
|------|------|---------|
| 真实连接 | DatabaseMCPService 连接真实 MySQL（docker-compose 已带），schema 查询走 information_schema，种子数据脚本灌入 loan_contract/customer_info | Agent 2 看到真实 schema；execute_sql 返回真实结果 |
| 只读强制三层 | ① sqlglot 解析 AST：仅允许单语句 SELECT（拒绝多语句/注释绕过/INTO OUTFILE/写函数）；② 数据库侧只读账号（readonly 用户无写权限）；③ 语句级超时 + 行数上限 | 注入/绕过用例测试全部被拒（含 `SELECT...;DROP`、`WITH...DELETE`） |
| 方言声明 | MySQL 验证链路真实化（test_verify 在 MySQL 容器跑）；Oracle/GaussDB 在文档声明为"适配器扩展点"，T002 数据源改 MySQL 或暂时隐藏 | T002 不再必然失败；README 说明方言支持矩阵 |
| 异步修复 | 同步 sqlite3 调用改 aiosqlite/异步驱动或 run_in_executor | 并发任务不阻塞事件循环 |

#### Day 8–9：真实向量检索（P0）

| 任务 | 内容 | 验收标准 |
|------|------|---------|
| 真实 embedding | 接入本地中文 embedding（如 BGE-small-zh / text2vec，离线可跑）或 OpenAI 兼容 embedding 端点；`embedding_provider` 配置真正生效 | 向量含语义信息；无语义查询也能召回（同义表达测试） |
| 检索融合 | 向量余弦通道 + 现有 bigram 文本打分融合排序；阈值可配 | 检索测试页可见两种得分 |
| 索引一致性 | 禁用/删除文档同步移除向量（修 rag.retrieve 不传 active_doc_ids 的 bug）；索引写加锁（文件锁或迁移 SQLite 存储） | 禁用后 Agent 1 检索不再召回；并发索引不丢数据 |
| 检索评测 | 10–20 条业务问答评测集 + Top-1/Top-5 命中率统计脚本 | 有可量化的检索质量基线 |

#### Day 10：配置外移 + 收口（P1）

| 任务 | 内容 | 验收标准 |
|------|------|---------|
| 租户动态化 | tenants 表真正启用，PRESET_TENANTS 迁移为种子数据；新增租户管理 API（创建/配置数据源与 AI 后端） | 不改代码可加租户 |
| 清理 | 删死代码（AgentRegistry/未用适配器）、配置与实现对账（faiss 等空头配置移除或实现）、Alembic 迁移引入、debug 默认 False | 代码库无死代码；迁移脚本可用 |
| Mock fail-fast | mock_mode 仅显式开启时生效；api_key 缺失且非 mock → 启动报错 | 配错 Key 立即暴露 |
| 深度健康检查 | /health 检查 DB 可写、向量目录可用、AI 后端连通（分级 ok/degraded/down） | 拔掉 DB 后 /health 变 down |
| 测试收口 | 六维校验/切片/分词纯函数单测；CI（GitHub Actions）跑 pytest + 冒烟；冒烟改临时目录隔离 | CI 绿；本地一键 pytest |
| 文档 | README 重写（边界声明更新为"L2 已完成项/仍为 Demo 裁剪项"两张清单）+ 架构图更新 | 评委看 README 无未声明裁剪 |

### 里程碑与风险

| 检查点 | 时间 | 标准 |
|--------|------|------|

| 两周时间紧张 | P0 项必须完成，P1 项（Day 10 部分）可顺延，优先保证 W1 全部 + Day 6–9 |
